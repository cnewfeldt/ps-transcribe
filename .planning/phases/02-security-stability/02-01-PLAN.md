  <action>
Apply four fixes to release-dmg.yml:

**SECR-01 (GH_TOKEN in clone URL, line ~142):**
Replace the token-embedded git clone with `gh repo clone`. The workflow already uses `gh release upload` so gh CLI is available. Replace:
```bash
git clone --branch gh-pages --single-branch "https://x-access-token:${GH_TOKEN}@github.com/OWNER/ps-transcribe.git" /tmp/gh-pages
```
With:
```bash
gh repo clone OWNER/ps-transcribe /tmp/gh-pages -- --branch gh-pages --single-branch
```
The `GH_TOKEN` env var is already set as `${{ github.token }}` which gh CLI reads automatically. Note: OWNER/ps-transcribe is a placeholder from Phase 1 -- preserve it as-is (per Phase 1 decision).

**SECR-05 (predictable keychain path, line ~47):**
Replace:
```bash
KEYCHAIN_FILE="/tmp/build-$$-$(date +%s).keychain-db"
```
With:
```bash
KEYCHAIN_FILE=$(mktemp /tmp/keychain.XXXXXX.keychain-db)
```

**SECR-07 (unpinned actions):**
Pin all `uses:` directives to commit SHAs. Before committing, verify SHAs by running:
```bash
git ls-remote https://github.com/actions/checkout refs/tags/v4
git ls-remote https://github.com/actions/upload-artifact refs/tags/v4
```
Use the SHA returned for each. Research suggests `34e114876b0b11c390a56381ad16ebd13914f8d5` for checkout v4 and `ea165f8d65b6e75b540449e92b4886f43607fa02` for upload-artifact v4 -- but verify at implementation time. Add a `# v4` comment after each SHA for readability:
```yaml
- uses: actions/checkout@{verified-sha}  # v4
```

**SECR-12 (2>/dev/null suppression, line ~90):**
Replace:
```bash
security delete-keychain "$KEYCHAIN_FILE" 2>/dev/null || true
```
With:
```bash
if ! security delete-keychain "$KEYCHAIN_FILE"; then
    echo "Warning: failed to delete keychain $KEYCHAIN_FILE -- may need manual cleanup" >&2
fi
```
Apply this pattern to ALL instances of `2>/dev/null || true` or `2>/dev/null` in the file.
